Internal Reference

FAQ Scraper — Full Workflow

How the lead magnet works end to end, including URL discovery, page selection, LLM extraction, and delivery.

00 — OVERVIEW

The Big Picture

A user submits competitor URLs + their email on the form page. Your server scrapes those sites, extracts every FAQ question and answer it can find, builds a CSV, and emails it to them. The whole thing runs in the background so the user does not wait on a loading screen.

1

User submits the form

Pastes competitor URLs + their email at faq.intelligentresources.app. LocalStorage is checked first — if they already submitted from this browser, they see the "Already Claimed" page without hitting the server.

2

POST /scrape — job created

Server checks submitted_emails.json. If already recorded → returns status: "already_submitted". Otherwise → records email, creates job ID, starts background scrape, returns { job_id, status: "processing" } instantly.

3

n8n picks up and polls

n8n receives job_id. If already_submitted → sends "already claimed" email and stops. Otherwise waits 10s, then polls GET /result/{job_id} every 10s until done.

4

Background scrape runs

For each URL: discover pages → categorise → select up to 15 → fetch content → send to LLM → extract Q&A → deduplicate → build CSV.

5

n8n delivers the email

Gets status: "done" with base64 CSV. Decodes it, attaches to Gmail, sends to user. If no FAQs found, sends a "no content found" email instead.

01 — URL DISCOVERY

Finding Pages to Scrape

Given a single URL like kynection.com.au, the scraper finds all pages worth checking using two sources simultaneously, then merges them.

Firecrawl Map

Crawls the site by following <a> links — up to 500 URLs including subdomains. Good at finding linked pages but misses pages not in the navigation.

Sitemap Parser

Fetches /sitemap.xml directly. Catches pages that exist but are not linked anywhere. Always runs alongside Firecrawl, never just a fallback.

Help subdomains: If Firecrawl finds help.kynection.com or docs.kynection.com, it runs a full separate map on that subdomain.
MadCap Flare: Some help centres render entirely in JavaScript — Firecrawl only finds 1 page. The scraper detects this, fetches /Data/HelpSystem.js, and follows the table-of-contents to pull all page URLs manually.

03 — PAGE SELECTION

Picking Which Pages to Scrape

Up to 15 pages per submitted URL are selected in priority order. Unused slots from a higher category flow down to the next.

  • 1stfaq Dedicated FAQ pages max 3
  • 2ndhelp Help/support/docs pages max 1
  • 3rdhome Homepage max 1
  • 4tharticle_index Blog/content index pages max 2
  • 5tharticle_post Individual blog posts max 5
  • 6thother Service, pricing, feature pages — shortest path first fill all remaining
EXAMPLE — kynection.com.au (no /faq page)
0 faq pages → 0 slots used.
1 help page → 1 slot.
1 home → 1 slot.
2 blog indexes → 2 slots.
5 blog posts → 5 slots (capped).
6 remaining slots → filled by "other" pages: /about, /pricing, /fleet-management, /integrations, /compliance, /contact.

Total: 15 pages scraped.
04 — CONTENT FETCHING

Getting the Text from Each Page

Every page is fetched two ways. Whichever produces more text wins.

Static HTTP (requests + BeautifulSoup)

Raw HTML before JavaScript runs. Catches CSS-hidden content like Webflow accordion answers (display:none) that a browser strips from the live DOM.

Firecrawl rawHtml (headless browser)

Full browser render. Catches JavaScript-rendered content (React/Vue SPAs) that does not exist in static HTML.

Max 60,000 characters per page are sent to the LLM. Content beyond that is truncated to avoid cutting off FAQ sections near the bottom of long pages.
05 — LLM EXTRACTION

Extracting Q&A Pairs with AI

Cleaned page text is sent to Claude Haiku 4.5 via OpenRouter. The prompt returns a JSON array of question/answer pairs.

THE EXACT PROMPT

You are a data extraction assistant. Given the text content of a web page, extract FAQ-style question and answer pairs.

Rules:
1. Extract EXPLICIT Q&A pairs — where a question is clearly written out (with or without a "?").
2. Also extract IMPLIED Q&A pairs — where a descriptive heading is directly followed by a paragraph that clearly explains it. Convert the heading into a natural question (e.g. "How we handle onboarding" → "How do you handle onboarding?"). Skip vague labels, CTAs, navigation text, or headings with no meaningful body paragraph.
3. For dedicated FAQ pages: extract every Q&A pair found.
4. For blog posts: extract explicit FAQ sections only. Return [] if none.
5. Keep answers concise — key info only, no marketing fluff. Max 500 characters per answer.
6. Return ONLY valid JSON. No markdown, no explanation, no code fences.

Return format:
[
  {"question": "What is X?", "answer": "X is..."},
  {"question": "How does Y work?", "answer": "Y works by..."}
]

If no extractable content is found, return: []

Page content:
---
[ page text — up to 60,000 characters ]
---
EXAMPLE 1 — Explicit FAQ page
Page text:
"Does Kynection work offline?
Yes. The mobile app stores data locally and syncs when connectivity is restored."

LLM returns:
[{"question": "Does Kynection work offline?", "answer": "Yes. The app stores data locally and syncs when back online."}]
EXAMPLE 2 — Implied Q&A from service page heading (new)
Page text:
"How we handle onboarding
Every new client gets a dedicated onboarding specialist for the first 30 days. We configure your workflows, train your team, and run a sign-off session before going live."

LLM returns:
[{"question": "How do you handle onboarding?", "answer": "Every new client gets a dedicated onboarding specialist for 30 days. We configure workflows, train your team, and run a sign-off before go-live."}]
EXAMPLE 3 — Heading the LLM will SKIP
Page text:
"Get Started Today
Book a free demo."

LLM returns: [] — CTA heading with no meaningful explanatory content.

07 — ONE PER PERSON GATE

How the One-Report-Per-Email Rule Works

A

LocalStorage check (browser)

On page load the browser checks for key ir_faq_submitted. If found → shows "Already Claimed" page immediately, no server request made.

B

Server-side email check

Even if localStorage is cleared (private browsing, new device), the server checks submitted_emails.json. If email is recorded → returns status: "already_submitted".

C

n8n sends "already claimed" email

n8n receives already_submitted → sends email telling the user their report was already sent → directs them to book a strategy call.

Where is submitted_emails.json stored? On the Coolify server filesystem, same directory as api.py. Persists across restarts. Would be wiped if the container volume is reset.
